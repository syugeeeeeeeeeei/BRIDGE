# v0.13.1 アーキテクチャ修正計画

## 1. コード側の欠陥

1. `src/bridge/bearing` がCollector、Sink、ファイル保存、集計を所有していた。
2. GATEがCollectorとSinkを生成し、観測実装を具体的に制御していた。
3. GATEが`graph.type=file`を直接読み込み、外部I/Oを所有していた。
4. TRAFFICと製品CLIがBEARINGの保存実装へ直接依存していた。
5. SinkのClose失敗が探索成功を破棄する可能性があった。
6. 依存規則を包括的に検査するarchitecture testが不足していた。

## 2. 文書側の欠陥

1. リポジトリ構成が旧`cmd/`構造のままだった。
2. GATEの観測契約注入に必要なBEARING契約依存が依存表に記載されていなかった。
3. GATEがファイルI/Oを所有しないことが明文化されていなかった。
4. BEARINGとULTRASOUNDの責務境界が、Collector／Sink単位で十分に具体化されていなかった。
5. 観測失敗時の探索結果保持方針が利用手順へ反映されていなかった。

## 3. 修正方針

- Collector、Sink、保存、集計、truncationをULTRASOUNDへ移す。
- BEARINGにはEvent、Observer、NullObserver、SafeObserverだけを残す。
- GATEはBEARINGのObserver契約だけを受け取り、CollectorやSinkを生成しない。
- graph fileの読込みはCLI等の入力アダプターでinline化してからGATEへ渡す。
- TRAFFICとCLIはULTRASOUNDを組み立て、GATEへObserverとして注入する。
- 観測Close失敗は探索結果を破棄せず、stderr警告またはbenchmark failure記録へ分離する。
- package依存表をarchitecture testで固定する。

## 4. 完了条件

- `src/bridge/bearing`にCollector／Sink／Recorder実装が存在しない。
- `src/bridge/traffic`がBEARINGを直接importしない。
- GATEがファイルを開かない。
- Trace有効・無効で探索結果とWorkが一致する。
- 観測書込み失敗でもRoute Resultが返る。
- `go test ./...`、`go test -race ./...`、`go vet ./...`が成功する。

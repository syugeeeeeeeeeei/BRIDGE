# BRIDGE v0.15.0 致命的欠陥修正チェックリスト

## 判定基準

本チェックリストは、ベンチマークで検出された「結果の意味論、証明、計測、研究データの信頼性を破壊する欠陥」を対象とする。単なる性能劣位は致命的欠陥には含めず、別途性能改善課題として扱う。

| ID | 致命的欠陥 | 修正要件 | 合格証拠 | 判定 |
|---|---|---|---|---|
| F-01 | Reachability が重み付き最短路を証明していないのに `optimality_proven=true` を返す | Reachability は常に最短路 Exact を主張しない。到達性証明と最適性証明を分離する | `TestReachabilityDoesNotClaimWeightedOptimality` | 達成 |
| F-02 | Reachability の到達不能・予算不足・キャンセルが排他的でない | `SearchCompleted`、`ReachabilityProven`、`TerminationStatus`、`ErrorCode` を終了理由ごとに整合させる | `TestReachabilityBudgetExhaustionIsNotAProof`、`TestReachabilityUnreachableProof` | 達成 |
| F-03 | GATE が内部 Solver の証明状態を捨て、`search_completed` から到達性証明を捏造する | 公開結果は内部証明を伝播し、発見経路だけは到達性証明として扱う | 全 GATE/TRAFFIC 統合テスト | 達成 |
| F-04 | TRAFFIC が `found` と `error_code` から証明状態を再推定する | Route/ExecuteOnce の公開証明フィールドをそのまま保存する | 全 TRAFFIC/HEALTHY 統合テスト | 達成 |
| F-05 | Reachability の Solver 境界時間が未設定で `timing_valid=false` になる | 正常終了・予算不足・キャンセルを含む全出口で正の Solver/Total/BOLTS 時間を記録する | Reachability 3テスト、修正後ベンチマーク126 Run | 達成 |
| F-06 | 矛盾した証明・計測値が研究結果へ無検証で混入する | TRAFFIC を fail-closed とし、最適性、到達性、予算終了、Timing の不変条件違反を拒否する | `TestValidateBenchmarkRunClaims*` | 達成 |
| F-07 | warmup でも trace/collector を有効化し、捨てるデータ生成が大規模ベンチマークを支配する | warmup の Observation を強制的に off とし、trace と collector を生成しない | シナリオ実行テスト、通常テスト、race test | 達成 |
| F-08 | 修正が並行実行時に破綻する可能性 | 全パッケージで race detector を通す | `go test -race -count=1 ./...` | 達成 |
| F-09 | 修正後の公開ベンチマーク結果が旧不正値を維持する可能性 | 同一126 Runを再実行し、Reachability Exact率0%、Timing有効率100%、失敗0件を確認する | `BRIDGE-v0.15.0-fixed-benchmark-result.json` | 達成 |

## 非対象

BRIDGE が Dijkstra より遅いこと、複数仮説による Work 増加は重大な性能課題だが、結果・証明・計測の意味論を偽る欠陥ではないため、本チェックリストの「致命的欠陥修正」には含めない。性能改善は探索戦略、仮説選択、epoch grant、Handoff 発火条件の別計画で扱う。
